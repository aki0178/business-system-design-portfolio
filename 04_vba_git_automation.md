# 04. VBA Git管理自動化ツール

> 保存トリガーでVBAソースをエクスポートし、自動コミットまで行う「開発基盤ツール」。
> 他の業務ツール（正規化／転記／可視化）の変更履歴とトレーサビリティ、バージョン管理によるコード品質の担保を支える土台として設計。

---

## 1. 概要

VBAソースコードを保存イベント起点で自動エクスポート・自動コミットする基盤ツール。本ポートフォリオの他3本（正規化／転記／可視化）を版管理面から支える土台として位置づけている。

---

## 2. 背景と課題

VBA開発には、一般的な開発環境にはない固有の制約がある。

- `.xlsm` / `.xlsb` はバイナリのため、Gitで差分を取っても**意味のある履歴にならない**
- 上記理由からExcelファイルをそのまま履歴として保存する必要があり、**ファイル管理が煩雑になりがち**である
- IDEがVBE固定で、外部エディタ・Lint・レビュー文化と接続しにくい
- 複数ブックを横断して開発していると、**どのブックのどのモジュールをいつ変更したか**が追えなくなる
- 「保存し忘れエクスポート」「コミット忘れ」によるコードのロスト事故が起きやすい

業務で扱う4本のツール（正規化／転記／可視化／本ツール）を継続的に保守するうえで、**ソースコードをテキストとして版管理する仕組み**が前提条件であった。

---

## 3. 解決アプローチ

「保存」という**既存の業務動作にフック**して、開発者が意識せずに版管理が回る状態を作る。

- 保存イベント (`Workbook_BeforeSave`) を起点にする
- VBAコンポーネントを型別にエクスポート（.bas / .cls / .frm）
- 同一トリガー内で `git add` → `git commit` まで完了させる
- 全工程の結果をイミディエイトウィンドウに出力し、後追い可能にする

---

## 4. アーキテクチャ

### 4.1 全体構造

```
┌─────────────────────────────┐
│ PERSONAL.XLSB               │
│  └ 自動モジュールエクスポート管理.bas │  ← インストーラー
│     ├ 自動エクスポート機能をインストール() │
│     └ 自動エクスポート機能をアンインストール() │
└─────────────┬───────────────┘
              │ 注入
              ▼
┌─────────────────────────────┐
│ 対象ブック (.xlsm)           │
│  └ ThisWorkbook              │
│     └ Workbook_BeforeSave    │  ← 保存のたびに実行
│        ├ VBComponents エクスポート │
│        ├ git add . / git commit  │
│        └ Debug.Print ログ出力    │
└─────────────────────────────┘
```

### 4.2 出力先のディレクトリ設計

```
C:\Users\xxx\works\           ← GIT_REPO_PATH（Gitリポジトリのルート）
└ コード\
   ├ ブックA\
   │   ├ Module1.bas
   │   └ Sheet1.cls
   └ ブックB\
       └ ...
```

ブック名でフォルダを分離することで、**複数ブックを単一リポジトリで管理しても名前衝突が起きない**。また、「コード」フォルダで管理することによって元ファイルとの重複事故を防いでいる。

---

## 5. 設計判断と意図

| 判断 | 採用した理由 | 検討した代替案 |
| --- | --- | --- |
| インストーラー方式（コードを文字列で注入） | 対象ブックごとに同じロジックを手で貼る運用を避けたい。配布も `インストール()` 一発で済む | アドイン化（.xlam）／対象ブックに直接コードを書く |
| `Workbook_BeforeSave` フック | 「保存」という既存動作に乗せれば、開発者の手数ゼロで運用が回る | リボン上のボタン化、定期実行 |
| `vbComp.Type` で拡張子を出し分け | 標準モジュール／クラス／フォームを正しい拡張子で出さないとVBEに再インポートできない | 一律 `.txt` 出力 |
| `chcp 932` を `cmd /c` 内で指定 | 版保存後の処理の際のコード内容（特に日本語）の文字化けを防ぐ | UTF-8 (`chcp 65001`)／PowerShell経由 |
| `Debug.Print` で全工程ログ出力 | サイレント失敗を防ぐ。Gitの `stderr` まで拾うことで「コミットされていないのに気づかない」事故を回避 | ログファイル出力／MsgBox通知 |
| `On Error GoTo ErrorHandler` → `Resume Next` | **保存処理そのものは絶対に止めない**。Excel自体の作業が止まらないようにするため | エラーをユーザーに通知して中断 |
| アンインストーラー同梱 | インストール後の予期しない動きで問題が発生した際、原因を取り除けるようにする。また、ブック配布の際にマクロを削除する手間を省く | .basに保存して都度インポート |

---

## 6. 処理フロー

### 6.1 インストール時

1. 対象ブックをアクティブにする
2. `自動エクスポート機能をインストール()` を実行
3. 対象ブックの `ThisWorkbook` モジュールに、保存トリガー用コードが文字列として書き込まれる
4. マクロ有効形式でブックを保存

### 6.2 保存時（自動実行）

1. `Workbook_BeforeSave` が走る
2. 出力先 `\コード\<ブック名>\` が存在しなければ作成
3. `VBComponents` をループし、型別に `.bas` / `.cls` / `.frm` でエクスポート
4. エクスポート結果をイミディエイトに出力
5. `cmd` 経由で `git add . && git commit -m "Update yyyy-MM-dd HH:mm"` を実行
6. Gitの `stdout` / `stderr` / `ExitCode` をイミディエイトに出力
7. エラーが出ても保存処理は継続

---

## 7. 学びと振り返り

- **「メタツール」をポートフォリオに含める価値**：このツール自体は表に出る成果物ではないが、他3本を「変更履歴の追えるコード」として残せたのは、この基盤を先に作っていたからだと感じている。設計を語るときに「土台から自分で整えた」と言えることの重みを、作ってから後で実感した。

- **既存動作へのフックは強い**：「新しい習慣をユーザーに強いない」のは、自分以外が使うツールでも、自分自身が使うツールでも、最重要の設計原則だと実感した。
- **可観測性の優先度を上げた**：最初は `MsgBox` で通知していたが、毎回の保存で出るのは邪魔である。`Debug.Print` に倒したことで「普段は静か、必要なときだけ見にいける」状態になった。

---
